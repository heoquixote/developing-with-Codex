# 5-1. codex exec와 CI/CD 자동화

## 목차

1. non-interactive mode 이해하기
2. 스크립트에서 Codex 호출하기
3. GitHub Actions에 연결하기
4. 반복 작업을 자동화 파이프라인으로 바꾸기
정리

Codex를 대화형 도구로만 쓰면 생산성은 올라가지만, 반복 작업은 여전히 사람의 손을 탑니다. `codex exec` 같은 비대화형 실행 방식이 중요한 이유는, 자주 반복되는 검토와 정리 작업을 스크립트와 파이프라인으로 이동시킬 수 있기 때문입니다.


## 1. non-interactive mode 이해하기

비대화형 모드의 핵심은 "질문에 답하며 사용하는 도구"를 "입력과 출력이 정의된 실행 단계"로 바꾸는 것입니다. 이 모드에서는 결과의 화려함보다 재현성과 일관성이 더 중요합니다.

따라서 프롬프트도 사람 친화적 설명보다 기계적 안정성이 우선됩니다. 입력 형식, 산출 형식, 실패 조건, 종료 기준이 분명해야 자동화에 잘 들어갑니다.


## 2. 스크립트에서 Codex 호출하기

스크립트에서 Codex를 호출할 때는 다음 세 가지를 먼저 정해야 합니다.

- 어떤 입력을 넘길 것인가
- 어떤 산출물을 받을 것인가
- 실패 시 파이프라인이 어떻게 반응해야 하는가

예를 들어 릴리스 노트 초안 생성, 변경 diff 요약, 문서 업데이트 제안, 테스트 실패 분석 초안 작성 같은 작업은 스크립트와 잘 맞습니다.


## 3. GitHub Actions에 연결하기

CI/CD와 연결할 때는 "무엇을 자동화해야 하는가"보다 "무엇은 아직 사람이 봐야 하는가"를 먼저 정해야 합니다. 모든 것을 자동화하려 하면 오히려 운영 리스크가 커집니다.

좋은 자동화 후보는 보통 다음과 같습니다.

- 변경 요약 생성
- 문서 누락 탐지
- PR 설명 초안 작성
- 실패 로그 분석 초안 작성

반면 배포 결정이나 보안 승인처럼 책임 경계가 명확해야 하는 영역은 사람 중심으로 남겨두는 편이 안전합니다.


## 4. 반복 작업을 자동화 파이프라인으로 바꾸기

자동화의 진짜 가치는 시간 절약보다 표준화에 있습니다. 사람이 매번 다르게 하던 작업을 동일한 형식과 기준으로 처리하게 만들면, 품질 편차가 줄고 팀의 운영 비용이 내려갑니다.

중요한 것은 "잘 되던 개인 습관"을 그대로 옮기는 것이 아니라, 팀이 이해하고 유지할 수 있는 단계로 단순화하는 것입니다.


## 정리

`codex exec`와 CI/CD 자동화는 Codex를 도구에서 시스템 구성요소로 바꾸는 첫 단계입니다. 여기서부터는 생산성 향상뿐 아니라 운영 설계의 문제가 함께 등장합니다.
