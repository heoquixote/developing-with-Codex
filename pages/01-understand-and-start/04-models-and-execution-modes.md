정리

Codex를 조금만 써보면 금방 알게 됩니다. 좋은 결과를 만드는 것은 프롬프트만의 문제가 아닙니다. 어떤 모델을 쓰는지, 로컬에서 바로 작업할지, 클라우드에 맡길지에 따라 체감이 꽤 달라집니다.

비유하면 이렇습니다. 모든 일을 같은 크기의 가방 하나에 넣고 다니면 불편합니다. 짧은 외출용 가방, 노트북 가방, 여행용 캐리어가 다 따로 있듯, 모델과 실행 방식도 일의 크기와 성격에 맞게 고르는 편이 훨씬 자연스럽습니다.

## 1. gpt-5.4와 gpt-5.4-mini를 언제 쓰는가

아주 단순하게 시작하면 됩니다.

- `gpt-5.4-mini`는 빠르게 여러 번 주고받을 때 좋습니다.
- `gpt-5.4`는 더 긴 맥락과 깊은 판단이 필요할 때 좋습니다.

쉽게 말해 `mini`는 가벼운 메모장, `gpt-5.4`는 오래 앉아서 정리하는 큰 책상에 가깝습니다.

실무에서는 아래처럼 나누면 초반에 편합니다.

- 빠른 코드 탐색, 문서 다듬기, 작은 수정은 `gpt-5.4-mini`
- 설계 비교, 다단계 디버깅, 복잡한 리팩터링 검토는 `gpt-5.4`

중요한 점은 "큰 모델이 항상 더 좋다"가 아니라, 실패 비용이 큰 작업일수록 더 안정적인 판단이 필요하다는 점입니다.

## 2. 더 무거운 작업에서 Codex 계열 모델을 고르는 기준

작업이 무거워질수록 아래 세 가지를 보면 됩니다.

- 여러 파일과 긴 문맥을 같이 봐야 하는가
- 중간 판단을 여러 번 이어가야 하는가
- 잘못 바꾸면 되돌리기 비용이 큰가

이 세 조건이 강할수록 더 깊게 생각하는 모델이 유리합니다. 반대로 이미 구조가 뚜렷하고 정답이 거의 보이는 작업이라면 가벼운 모델로도 충분한 경우가 많습니다.

초심자 기준으로는 이렇게 기억하셔도 됩니다. "빨리 확인하는 일은 가볍게, 비싸게 틀릴 수 있는 일은 신중하게."

## 3. Local 작업과 Cloud 작업의 차이

로컬 작업은 지금 내 앞의 저장소와 바로 붙어 있는 방식입니다. 파일을 읽고, 테스트를 돌리고, 수정 결과를 바로 확인할 수 있다는 점이 큰 장점입니다.

클라우드 작업은 시간이 오래 걸리거나 병렬로 돌릴 가치가 있는 일을 따로 맡길 때 좋습니다.

비유하면 아래처럼 생각하시면 쉽습니다.

- 로컬은 지금 내 손으로 바로 수리하는 작업대입니다.
- 클라우드는 오래 걸리는 작업을 맡겨두는 별도 작업실입니다.

그래서 현재 브랜치에서 세밀하게 고칠 때는 로컬이 자연스럽고, 긴 탐색이나 여러 아이디어 비교는 클라우드가 더 어울립니다.

## 4. 비용·속도·정확도의 균형

이 세 가지는 동시에 모두 최고로 맞추기 어렵습니다. 보통 속도를 높이면 깊이가 줄고, 정확도를 높이려면 더 많은 추론과 검증이 들어가 비용이 올라갑니다.

그래서 실무에서는 단계별로 우선순위를 바꾸는 편이 좋습니다.

- 탐색 단계에서는 속도를 우선합니다.
- 설계 결정 단계에서는 정확도를 우선합니다.
- 구현 단계에서는 변경 범위를 줄여 비용을 관리합니다.
- 검증 단계에서는 모델보다 테스트와 diff를 더 믿습니다.

결국 비용을 줄이는 가장 좋은 방법은 무조건 작은 모델을 쓰는 것이 아닙니다. 큰 작업을 작고 검증 가능한 단위로 나누는 것입니다. 무거운 짐도 나눠 들면 훨씬 안전한 것과 같습니다.

## 정리

모델 선택은 취향 문제가 아니라 작업 설계의 일부입니다. 어떤 모델이 더 똑똑한가보다, 지금 하는 일이 빠른 왕복이 필요한지, 깊은 판단이 필요한지를 먼저 보는 편이 훨씬 실용적입니다. Codex를 잘 쓰는 사람은 프롬프트만 잘 쓰는 사람이 아니라, 모델과 실행 모드를 일의 성격에 맞게 배치하는 사람입니다.
