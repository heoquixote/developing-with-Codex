# 1-1. Codex를 어떤 도구로 봐야 하는가

Codex를 제대로 쓰려면 먼저 이 도구를 어떤 범주로 이해할지부터 정해야 합니다. 많은 사용자가 처음에는 "코드를 잘 써주는 자동완성 도구"로 접근합니다. 이 관점은 틀리지는 않지만 충분하지도 않습니다. Codex의 진짜 강점은 한 줄 제안을 잘하는 데보다, 목표를 받아 탐색하고, 수정하고, 검증하고, 결과를 보고하는 작업 흐름을 끝까지 이어가는 데 있습니다.

## 목차

## 코딩 보조도구와 코딩 에이전트의 차이

코딩 보조도구는 대개 현재 편집 중인 파일 근처에서 다음 토큰이나 다음 코드를 제안합니다. 반면 코딩 에이전트는 더 넓은 범위의 목표를 받아 여러 파일을 읽고, 필요한 명령을 실행하고, 변경 결과를 설명합니다. 차이는 "코드를 얼마나 잘 쓰는가"보다 "작업 단위를 어디까지 맡길 수 있는가"에 있습니다.

실무에서는 이 차이가 매우 큽니다. 자동완성은 개발자의 손을 빠르게 하지만, 에이전트는 개발자의 작업 큐 자체를 줄입니다. 예를 들어 "이 에러의 원인을 찾고 회귀 테스트까지 추가해줘" 같은 요청은 자동완성의 영역이 아니라 에이전트의 영역입니다.

## 로컬 실행, IDE 연동, 클라우드 위임의 개념

Codex를 사용하는 실행 방식은 크게 세 가지로 이해하면 됩니다.

- 로컬 실행은 현재 작업 중인 저장소와 터미널 맥락을 가장 깊게 활용하는 방식입니다.
- IDE 연동은 편집 중인 파일과 주변 컨텍스트를 빠르게 주고받으며 페어 프로그래밍하듯 쓰는 방식입니다.
- 클라우드 위임은 오래 걸리거나 병렬화 가치가 큰 작업을 분리해 맡기는 방식입니다.

셋의 차이는 단순히 UI가 아니라 비용 구조와 피드백 속도, 그리고 적합한 업무 유형입니다. 로컬은 디버깅과 세밀한 수정을 잘하고, IDE는 빠른 왕복에 강하며, 클라우드는 긴 실행과 병렬 실험에 강합니다.

## Codex가 잘하는 일과 취약한 일

Codex가 잘하는 일은 구조가 있는 작업입니다. 예를 들어 코드베이스 설명, 반복 패턴 정리, 작은 기능 추가, 테스트 보강, 문서 초안 작성, 리뷰 포인트 식별 같은 일은 매우 잘합니다. 이유는 목표와 제약이 명확할수록 탐색과 실행 경로가 안정되기 때문입니다.

반대로 취약한 일도 분명합니다. 요구사항 자체가 모호한 상태에서 큰 설계 결정을 대신 내려야 하거나, 검증 수단 없이 광범위한 리팩터링을 한 번에 수행해야 하거나, 조직 문맥이 전혀 공유되지 않은 정책 판단을 해야 하는 상황은 흔들리기 쉽습니다.

원리는 간단합니다. Codex는 추론 능력만으로 일하는 것이 아니라 컨텍스트와 검증 가능성에 크게 의존합니다. 따라서 잘 쓰려면 "정답을 맞혀라"보다 "탐색 경로와 검증 기준을 제공하라"로 접근해야 합니다.

## 이 책의 실습 방식과 예제 저장소 소개

이 책은 설명형 책이면서 동시에 반복 실습서입니다. 같은 예제 저장소를 여러 장에서 재사용하면서, 탐색, 구현, 디버깅, 리팩터링, 리뷰, 자동화까지 연결합니다. 이렇게 해야 독자가 도구 기능을 장식처럼 배우지 않고, 하나의 코드베이스에 대해 점점 더 깊은 협업을 경험할 수 있습니다.

실습에서는 작은 작업부터 시작합니다. 첫 세션에서는 코드베이스 설명 요청, 작은 버그 수정, diff 확인, 테스트 실행처럼 짧고 명확한 루프만 다룹니다. 이후 장이 진행될수록 같은 저장소에 기능 추가, PR 분리, AGENTS.md 적용, 자동화 파이프라인 설계 같은 요소를 얹습니다.

## 정리

Codex를 단순한 코드 생성기로 보면 결과를 평가할 기준이 "코드를 잘 뽑았는가"에 머무릅니다. 하지만 협업형 에이전트로 보면 기준이 달라집니다. 탐색을 잘했는가, 안전하게 바꿨는가, 검증까지 했는가, 인간이 검토하기 쉬운 결과를 남겼는가가 더 중요해집니다. 이 책 전체는 바로 그 관점 위에서 전개됩니다.
