정리

Codex를 처음 보면 많은 분이 "코드를 좀 더 잘 써주는 자동완성"이라고 생각합니다. 완전히 틀린 말은 아니지만, 그 설명만으로는 부족합니다. 자동완성이 운전 중 내비게이션의 "다음 코너에서 우회전" 같은 도움이라면, Codex는 옆자리에 앉아 지도를 같이 보고, 길도 찾고, 실수한 길도 다시 잡아주는 동료에 더 가깝습니다.

이 차이를 먼저 이해해두면 이후 실습이 훨씬 편해집니다. Codex에게 무엇을 기대해야 하고, 무엇은 아직 사람이 직접 판단해야 하는지가 더 분명해지기 때문입니다.

## 1. 코딩 보조도구와 코딩 에이전트의 차이

코딩 보조도구는 보통 지금 보고 있는 파일 근처에서 다음 코드를 추천합니다. 말 그대로 "손을 빠르게 해주는 도구"입니다. 반면 코딩 에이전트는 더 큰 목표를 받아 여러 파일을 읽고, 필요한 명령을 실행하고, 변경 결과를 설명하는 쪽에 가깝습니다.

비유하면 차이가 더 쉽습니다.

- 코딩 보조도구는 키보드 옆의 빠른 단축키와 비슷합니다.
- 코딩 에이전트는 "이 일 먼저 해보고, 결과 알려줘"라고 맡길 수 있는 보조 작업자와 비슷합니다.

실무에서 이 차이는 꽤 큽니다. 자동완성은 한 줄을 빨리 쓰게 해주지만, 에이전트는 "로그인 오류 원인 찾아보기", "관련 테스트 위치 찾기", "작은 버그 수정하고 검증하기"처럼 작업 덩어리 자체를 줄여줍니다.

## 2. 로컬 실행, IDE 연동, 클라우드 위임의 개념

Codex를 쓰는 방식은 크게 세 가지로 이해하면 쉽습니다.

- 로컬 실행은 내 책상 위에서 바로 작업하는 방식입니다.
- IDE 연동은 편집기 옆에서 바로 대화하는 방식입니다.
- 클라우드 위임은 시간이 좀 걸리는 일을 다른 작업 공간에 맡겨두는 방식입니다.

조금 더 쉬운 비유를 쓰면 이렇습니다.

- 로컬은 주방에서 직접 칼과 도마를 잡고 요리하는 느낌입니다.
- IDE는 조리대 옆에서 "이 재료는 어떻게 썰까요?"를 바로 묻는 느낌입니다.
- 클라우드는 오래 끓여야 하는 냄비를 다른 화구에 올려두는 느낌입니다.

셋 중 무엇이 더 좋다기보다, 어떤 일에 더 잘 맞는지가 다릅니다. 디버깅과 테스트처럼 바로 확인해야 하는 일은 로컬이 편하고, 파일 하나를 보며 빠르게 고치는 일은 IDE가 편합니다. 오래 걸리는 탐색이나 병렬 실험은 클라우드가 더 자연스럽습니다.

## 3. Codex가 잘하는 일과 취약한 일

Codex는 구조가 있는 일을 잘합니다. 목적과 범위가 분명할수록 길을 잘 찾습니다. 예를 들면 아래 같은 일입니다.

- 코드베이스 구조 설명
- 관련 파일 찾기
- 작은 기능 추가
- 테스트 보강
- 문서 초안 작성
- 리뷰 포인트 정리

반대로 아래 같은 일은 흔들리기 쉽습니다.

- 요구사항 자체가 아직 흐린 상태에서 큰 설계를 대신 정하는 일
- 검증 없이 넓은 범위를 한 번에 뒤집는 일
- 조직 정책이나 팀 맥락이 거의 공유되지 않은 판단

이유도 어렵지 않습니다. Codex는 마술사가 아니라, 힌트와 검증 기준이 있을수록 강해지는 도구이기 때문입니다. 지도와 체크포인트가 있으면 잘 달리지만, 목적지도 없고 표지판도 없으면 사람처럼 헤맬 수 있습니다.

## 4. 이 책의 실습 방식과 예제 저장소 소개

이 책은 기능 목록을 소개하는 책이 아니라, 같은 예제 저장소를 여러 번 다시 방문하면서 점점 더 깊게 협업하는 책입니다. 처음에는 코드베이스를 읽고, 다음에는 작은 버그를 고치고, 그다음에는 기능을 추가하고, 나중에는 자동화까지 연결합니다.

이 방식이 좋은 이유는 독자가 "기능 하나 배웠다"에서 끝나지 않기 때문입니다. 같은 저장소를 반복해서 다루면, Codex를 단발성 도구가 아니라 함께 일하는 파트너처럼 느끼게 됩니다.

처음에는 아주 작은 루프만 다룹니다. 설명 요청, 작은 수정, diff 확인, 테스트 실행 정도면 충분합니다. 처음부터 거대한 기능을 맡기지 않는 이유도 여기에 있습니다. 수영을 배울 때 바로 먼바다로 나가지 않는 것과 같습니다.

## 정리

Codex를 단순한 코드 생성기로 보면 "코드를 잘 뽑았는가"만 보게 됩니다. 하지만 협업형 에이전트로 보면 기준이 달라집니다. 어디를 읽었는가, 얼마나 안전하게 바꿨는가, 검증까지 했는가, 사람이 검토하기 쉬운 결과를 남겼는가가 더 중요해집니다.

이 책은 바로 그 관점에서 출발합니다. Codex를 잘 쓰는 것은 버튼을 잘 누르는 일이 아니라, 일을 잘 나누고 잘 검토하는 법을 배우는 일에 가깝습니다.
