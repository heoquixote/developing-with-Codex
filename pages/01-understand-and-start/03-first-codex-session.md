# 1-3. 첫 번째 Codex 세션

첫 세션의 목표는 큰 기능을 완성하는 것이 아닙니다. "탐색 -> 변경 -> 검증 -> 검토"라는 협업 루프를 몸으로 익히는 것이 목표입니다. 이 루프를 한 번 제대로 경험하면, 이후의 큰 작업도 같은 패턴으로 다룰 수 있습니다.

## 목차

- 코드베이스 설명 요청하기
- 바로 써보는 첫 프롬프트
- 작은 버그 하나 수정시키기
- 변경 diff 읽기
- 테스트 실행과 결과 검토
- 자주 하는 실수
- 첫 세션 체크리스트
- 정리

## 코드베이스 설명 요청하기

처음에는 구현을 시키기보다 설명을 시키는 것이 좋습니다. 이유는 간단합니다. 지금 가진 컨텍스트가 부족하기 때문입니다. 먼저 Codex에게 이 저장소의 구조, 진입점, 핵심 모듈, 테스트 위치를 설명하게 하면, 사람이 봐야 할 지형도가 빠르게 잡힙니다.

좋은 첫 요청은 보통 다음 정보를 포함합니다.

- 무엇을 이해하고 싶은지
- 어떤 범위까지만 설명하면 되는지
- 어떤 형식으로 정리해주길 원하는지

예를 들어 "로그인 흐름을 추적해 관련 파일과 호출 순서를 설명해줘"는 좋고, "이 프로젝트 설명해줘"는 너무 넓습니다.

## 바로 써보는 첫 프롬프트

첫 세션에서는 긴 배경 설명보다 범위를 좁힌 요청이 더 좋습니다. 아래 정도면 충분히 좋은 시작점입니다.

```text
이 저장소에서 사용자 로그인 흐름만 추적해주세요.
다음 형식으로 답해주세요.
1. 진입 파일
2. 핵심 함수 호출 순서
3. 관련 테스트 파일
4. 먼저 읽어야 할 파일 3개
```

작은 버그 수정 요청도 같은 원리입니다.

```text
`src/utils/date.ts`에서 null 입력 시 예외가 나는 버그를 수정해주세요.
범위는 이 파일과 관련 테스트만으로 제한해주세요.
변경 후 diff 관점에서 왜 안전한지도 짧게 설명해주세요.
```

핵심은 "무엇을", "어디까지", "어떤 형식으로"를 동시에 주는 것입니다. 이 세 가지가 있어야 첫 응답 품질이 안정됩니다.

## 작은 버그 하나 수정시키기

첫 수정은 반드시 작아야 합니다. 버튼 레이블 오타, null 처리 누락, 날짜 포맷 오류, 조건문 분기 실수처럼 재현이 쉬운 문제부터 다루는 편이 좋습니다. 이렇게 해야 에이전트의 수정 능력을 평가하기보다, 협업 방식 자체를 연습하는 데 집중할 수 있습니다.

작은 버그를 고를 때의 기준은 세 가지입니다.

- 재현이 명확할 것
- 기대 결과가 분명할 것
- 영향 범위가 좁을 것

첫 세션에서 흔한 실수는 너무 빨리 큰 문제를 맡기는 것입니다. 그러면 결과가 나빠서가 아니라, 사용자 자신이 검토할 수 없는 범위가 되어 학습 효과가 떨어집니다.

## 변경 diff 읽기

Codex와 협업할 때 diff 읽기는 선택이 아니라 핵심 기술입니다. 좋은 에이전트 활용은 "얼마나 많이 자동화했는가"가 아니라 "얼마나 적은 비용으로 검토 가능한 결과를 받았는가"로 판단해야 합니다.

diff를 볼 때는 다음 순서로 보면 좋습니다.

1. 어떤 파일이 바뀌었는지 본다.
2. 변경이 요청 범위를 벗어나지 않았는지 본다.
3. 핵심 로직 변경이 있는 줄을 자세히 읽는다.
4. 테스트와 문서 변경이 함께 들어갔는지 본다.

이 순서를 지키면 코드를 한 줄씩 다 읽지 않아도 큰 위험을 빠르게 걸러낼 수 있습니다.

예를 들어 아래처럼 보면 됩니다.

```diff
diff --git a/src/utils/date.ts b/src/utils/date.ts
-export function formatDate(value: string) {
-  return dayjs(value).format("YYYY-MM-DD");
+export function formatDate(value?: string | null) {
+  if (!value) return "";
+  return dayjs(value).format("YYYY-MM-DD");
 }
```

이 diff에서 사람이 봐야 할 포인트는 세 가지입니다.

- 입력 타입이 넓어졌는지
- null 방어 로직이 기대한 정책과 맞는지
- 빈 문자열 반환이 호출부 계약을 깨지 않는지

## 테스트 실행과 결과 검토

수정이 끝났다면 반드시 검증이 따라와야 합니다. 테스트 실행은 에이전트를 믿지 못해서가 아니라, 신뢰를 구조로 만드는 과정입니다. 사람이 직접 코드를 짰을 때도 테스트를 돌려야 하듯, 에이전트가 작업했을 때는 오히려 더 필요합니다.

결과를 볼 때는 "통과/실패"만 보지 말고 다음까지 확인합니다.

- 어떤 테스트가 실제로 실행됐는지
- 변경과 직접 관련된 검증이 포함됐는지
- 실패 시 원인이 환경 문제인지 코드 문제인지
- 새 회귀 테스트가 필요한지

첫 세션에서 여기까지 경험하면, 이후 장에서 기능 추가나 디버깅을 할 때도 같은 루프를 반복하면 됩니다.

예를 들어 결과 검토는 아래처럼 단순 명확해야 합니다.

```bash
$ npm test

 PASS  src/utils/date.test.ts
  formatDate
    ✓ returns empty string for null input
    ✓ formats yyyy-mm-dd input correctly

Test Suites: 1 passed, 1 total
Tests:       2 passed, 2 total
```

여기서 중요한 것은 "통과했다"보다 "이번 수정과 직접 관련된 검증이 실제로 돌았는가"입니다. 테스트가 넓게 많이 돌았더라도, 관련 회귀 테스트가 없으면 신뢰는 낮습니다.

## 자주 하는 실수

- 첫 세션부터 큰 기능 추가를 시킵니다.
- 저장소 설명 요청을 너무 넓게 던집니다.
- diff를 보기 전에 결과 설명만 믿고 넘어갑니다.
- 테스트 성공 여부만 보고 어떤 테스트가 돌았는지는 확인하지 않습니다.

첫 세션은 성능 평가가 아니라 협업 루프 학습입니다. 범위를 좁히지 않으면 실습의 핵심이 흐려집니다.

## 첫 세션 체크리스트

- 저장소 설명 요청의 범위를 명확히 적었습니다.
- 수정 범위를 한두 파일 수준으로 제한했습니다.
- diff에서 변경 파일 수와 영향 범위를 확인했습니다.
- 관련 테스트 실행 결과를 직접 검토했습니다.

## 정리

첫 세션의 성패는 얼마나 많은 코드를 바꿨느냐가 아닙니다. 코드베이스를 읽고, 작은 문제를 안전하게 고치고, diff와 테스트로 검증하는 흐름을 익혔느냐가 더 중요합니다. Codex와의 협업은 이 작은 루프를 얼마나 안정적으로 반복하느냐에 달려 있습니다.
