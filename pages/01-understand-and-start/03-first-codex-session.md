정리

첫 세션의 목표는 대단한 기능을 완성하는 것이 아닙니다. 자전거를 처음 탈 때도 바로 장거리 라이딩을 가지 않듯, 첫 세션도 아주 짧은 루프를 익히는 데 집중하면 됩니다. 이 장에서 익힐 루프는 단순합니다. `읽기 -> 바꾸기 -> 확인하기 -> 검토하기`입니다.

이 한 바퀴만 제대로 돌아가면 이후의 큰 작업도 훨씬 덜 무섭습니다. 반대로 첫 세션에서 너무 큰 일을 맡기면, Codex 실력보다 내가 지금 무엇을 검토해야 하는지가 더 흐려지기 쉽습니다.

## 1. 코드베이스 설명 요청하기

처음에는 구현보다 설명을 시키는 편이 좋습니다. 처음 보는 동네에 가면 바로 집을 짓지 않고 먼저 지도를 보는 것과 같습니다. Codex에게 저장소 구조, 진입점, 핵심 모듈, 테스트 위치를 설명하게 하면 사람이 봐야 할 지형도가 빨리 잡힙니다.

좋은 첫 요청은 보통 세 가지를 담습니다.

- 무엇을 이해하고 싶은지
- 어디까지 설명하면 되는지
- 어떤 형식으로 정리해주면 좋은지

예를 들어 "로그인 흐름만 추적해서 관련 파일과 호출 순서를 설명해줘"는 좋습니다. 반대로 "이 프로젝트 설명해줘"는 너무 넓어서 초심자에게 오히려 더 막막할 수 있습니다.

## 2. 바로 써보는 첫 프롬프트

첫 세션에서는 긴 배경 설명보다 범위를 좁힌 요청이 더 좋습니다. 아래 정도면 충분히 좋은 시작점입니다.

```text
이 저장소에서 사용자 로그인 흐름만 추적해주세요.
다음 형식으로 답해주세요.
1. 진입 파일
2. 핵심 함수 호출 순서
3. 관련 테스트 파일
4. 먼저 읽어야 할 파일 3개
```

작은 버그 수정 요청도 같은 방식으로 줄 수 있습니다.

```text
`src/utils/date.ts`에서 null 입력 시 예외가 나는 버그를 수정해주세요.
범위는 이 파일과 관련 테스트만으로 제한해주세요.
변경 후 diff 기준으로 왜 안전한지도 짧게 설명해주세요.
```

핵심은 세 가지입니다. "무엇을", "어디까지", "어떤 형식으로". 이 세 가지만 분명해도 첫 응답 품질이 훨씬 안정됩니다.

## 3. 작은 버그 하나 수정시키기

첫 수정은 반드시 작아야 합니다. 버튼 문구 오타, null 처리 누락, 날짜 포맷 오류처럼 재현이 쉬운 문제부터 고르는 편이 좋습니다.

이유는 간단합니다. 첫 세션은 Codex 실력 시험이 아니라 협업 연습이기 때문입니다. 처음 수영을 배울 때 숨쉬기부터 익히듯, 여기서도 작은 변경을 안전하게 다루는 감각을 먼저 익히는 편이 좋습니다.

작은 버그를 고를 때는 아래 세 가지를 보면 됩니다.

- 재현이 명확한가
- 기대 결과가 분명한가
- 영향 범위가 좁은가

## 4. 변경 diff 읽기

Codex와 협업할 때 diff 읽기는 선택이 아니라 핵심 기술입니다. 결과 설명만 듣고 넘어가는 것은 영수증을 보지 않고 계산을 끝내는 것과 비슷합니다. 대체로 그럴듯해 보여도, 어디서 무엇이 바뀌었는지 직접 확인해야 안심할 수 있습니다.

diff를 볼 때는 아래 순서로 보면 편합니다.

1. 어떤 파일이 바뀌었는지 봅니다.
2. 요청 범위를 벗어난 파일이 없는지 봅니다.
3. 핵심 로직이 바뀐 줄을 자세히 봅니다.
4. 관련 테스트가 같이 들어갔는지 봅니다.

예를 들어 아래처럼 볼 수 있습니다.

```diff
diff --git a/src/utils/date.ts b/src/utils/date.ts
-export function formatDate(value: string) {
-  return dayjs(value).format("YYYY-MM-DD");
+export function formatDate(value?: string | null) {
+  if (!value) return "";
+  return dayjs(value).format("YYYY-MM-DD");
 }
```

이럴 때는 코드가 멋져 보이는지보다 아래 세 가지를 먼저 보면 됩니다.

- 입력 타입이 넓어졌는가
- null 방어가 기대한 정책과 맞는가
- 빈 문자열 반환이 다른 곳을 깨뜨리지 않는가

## 5. 테스트 실행과 결과 검토

수정이 끝났다면 반드시 검증이 따라와야 합니다. 테스트는 Codex를 못 믿어서 돌리는 것이 아니라, 신뢰를 눈으로 확인하는 절차입니다. 요리할 때 마지막으로 간을 보는 것과 비슷합니다.

결과를 볼 때는 "통과했다"만 보지 말고 아래도 함께 확인합니다.

- 어떤 테스트가 실제로 실행됐는가
- 이번 수정과 직접 관련된 검증이 포함됐는가
- 실패했다면 환경 문제인가 코드 문제인가
- 새 회귀 테스트가 필요한가

예를 들어 결과는 아래처럼 단순하고 명확해야 좋습니다.

```bash
$ npm test

 PASS  src/utils/date.test.ts
  formatDate
    ✓ returns empty string for null input
    ✓ formats yyyy-mm-dd input correctly

Test Suites: 1 passed, 1 total
Tests:       2 passed, 2 total
```

여기서 중요한 것은 초록색 글자 자체가 아니라, "이번 수정과 직접 연결된 테스트가 실제로 돌았는가"입니다.

## 6. 자주 하는 실수

- 첫 세션부터 큰 기능 추가를 시킵니다.
- 저장소 설명 요청을 너무 넓게 던집니다.
- diff를 보기 전에 결과 설명만 믿고 넘어갑니다.
- 테스트 성공 여부만 보고 어떤 테스트가 돌았는지는 확인하지 않습니다.

첫 세션은 속도전이 아니라 감각 익히기입니다. 여기서 범위를 좁혀두면 이후 장들이 훨씬 쉬워집니다.

## 7. 첫 세션 체크리스트

- 저장소 설명 요청 범위를 명확히 적었습니다.
- 수정 범위를 한두 파일 수준으로 제한했습니다.
- diff에서 변경 파일 수와 영향 범위를 확인했습니다.
- 관련 테스트 실행 결과를 직접 검토했습니다.

## 정리

첫 세션의 성패는 얼마나 많은 코드를 바꿨느냐가 아닙니다. 코드를 읽고, 작은 문제를 안전하게 고치고, diff와 테스트로 검증하는 흐름을 익혔느냐가 더 중요합니다. Codex와의 협업은 결국 이 작은 루프를 얼마나 안정적으로 반복하느냐에 달려 있습니다.
