정리

좋은 공구가 있어도 작업대가 흔들리면 결과를 믿기 어렵습니다. Codex도 비슷합니다. 초반에는 "Codex가 틀렸다"와 "내 환경이 아직 안 갖춰졌다"가 자주 섞여 보입니다. 그래서 실습 초반에는 똑똑한 요청보다 먼저, 흔들리지 않는 바닥을 만드는 일이 중요합니다.

쉽게 말하면 이 장은 집을 짓기 전 바닥을 고르게 다지는 과정입니다. 여기서 기준이 잘 잡혀 있으면 이후 장에서 Codex가 잘했는지, 환경이 문제였는지 구분하기 쉬워집니다.

## 1. Git 저장소와 브랜치 전략 준비

Codex와 협업할 때 Git은 단순한 백업이 아닙니다. 안전벨트에 가깝습니다. 작은 단위로 브랜치를 나누고, diff를 빠르게 검토하고, 필요하면 쉽게 되돌아갈 수 있어야 안심하고 실습할 수 있습니다.

처음에는 복잡한 Git 전략이 필요하지 않습니다. 아래 세 가지만 지켜도 충분합니다.

- 메인 브랜치에서는 바로 작업하지 않습니다.
- 기능 추가, 버그 수정, 문서화는 되도록 따로 분리합니다.
- 한 브랜치에는 하나의 목적만 담습니다.

이 규칙이 중요한 이유는 Codex가 실수할 수 있어서만이 아닙니다. 사람이 검토하기 쉬운 단위로 일을 잘게 나누기 위해서입니다. 장바구니에 물건을 한꺼번에 다 담아놓으면 계산이 어려운 것처럼, 변경도 한 번에 너무 많이 섞이면 검토가 어려워집니다.

## 2. CLI, IDE 확장, Codex app의 역할 구분

처음 쓰는 분들은 "셋 중 뭐가 정답인가요?"를 많이 묻습니다. 하지만 정답 하나를 찾기보다, 각 도구를 어디에 쓰면 편한지 감을 잡는 편이 훨씬 중요합니다.

- CLI는 저장소 전체를 훑고, 명령을 실행하고, 테스트를 확인할 때 좋습니다.
- IDE 확장은 지금 보고 있는 파일을 중심으로 빠르게 묻고 고칠 때 좋습니다.
- Codex app은 작업을 나눠서 관리하거나 병렬로 굴릴 때 좋습니다.

비유하면 이렇습니다.

- CLI는 공구함 전체를 열어두는 작업대입니다.
- IDE는 손에 쥔 드라이버처럼 지금 만지는 부품에 집중하게 해줍니다.
- Codex app은 여러 작업 상자를 따로 놓고 관리하는 선반과 비슷합니다.

어떤 요청이든 한 도구만 고집할 필요는 없습니다. 예를 들어 구조를 넓게 파악할 때는 CLI가 편하고, 문구를 바로 수정할 때는 IDE가 편합니다.

## 3. Python/Node 테스트 환경 준비

테스트 환경은 선택사항이 아니라 기준점입니다. Codex가 뭔가를 바꿨을 때, 그 변경이 괜찮은지 판단하는 가장 현실적인 방법이 테스트이기 때문입니다.

Python 프로젝트라면 최소한 아래 세 가지를 확인하면 됩니다.

- 가상환경 또는 의존성 관리 방식
- 테스트 실행 명령
- 린트나 포맷 검사 명령

Node 프로젝트도 생각은 같습니다.

- 패키지 설치 명령
- `test`, `lint`, `build` 같은 기본 스크립트
- 필요한 런타임 버전

이 부분을 왜 먼저 보느냐면, 테스트가 없는 실습은 정답지가 없는 문제집과 비슷하기 때문입니다. 풀기는 했는데 맞았는지 틀렸는지 바로 알기 어렵습니다.

## 4. 바로 따라 하는 최소 환경 세팅

처음부터 완벽한 환경을 만들 필요는 없습니다. 초반에는 "최소한 여기까지는 실행된다"는 선만 만들어도 충분합니다.

```bash
git clone https://github.com/<your-org>/<sample-repo>.git
cd <sample-repo>
git switch -c chore/codex-first-session
```

Python 예제라면 보통 아래 흐름으로 시작합니다.

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
pytest
```

Node 예제라면 아래처럼 맞춰도 좋습니다.

```bash
npm install
npm test
npm run lint
```

중요한 것은 명령을 많이 외우는 것이 아닙니다. "어느 명령이 성공하면 환경 준비가 끝났다고 볼 것인가"를 정하는 것입니다. 신발끈을 묶고 출발선에 서는 느낌이라고 보시면 됩니다.

## 5. 샘플 프로젝트 내려받기

책 실습은 같은 예제 저장소를 여러 장에서 반복해서 쓰는 방식이 가장 좋습니다. 그래야 한 장에서는 구조를 읽고, 다음 장에서는 기능을 추가하고, 또 다음 장에서는 버그를 잡는 흐름이 자연스럽게 이어집니다.

좋은 샘플 저장소는 너무 단순해도 안 되고, 너무 거대해도 안 됩니다. 초심자에게는 "작지만 실제 같은" 저장소가 가장 좋습니다.

- Python 또는 Node 중 적어도 하나의 테스트가 있어야 합니다.
- 문서가 완벽하지 않아야 합니다.
- 작은 개선 포인트와 버그가 조금은 있어야 합니다.
- 구조가 너무 크지 않아야 합니다.

연습용 자전거가 너무 고급 레이싱 바이크여도 어렵고, 너무 장난감 같아도 감이 안 오는 것과 비슷합니다.

## 6. 자주 하는 실수

- 메인 브랜치에서 바로 실습을 시작합니다.
- 테스트 명령을 확인하지 않고 수정부터 시킵니다.
- Python과 Node 중 무엇이 기준 환경인지 정하지 않습니다.
- 저장소는 받았지만 실행 기준 명령을 적어두지 않습니다.

이 실수들이 위험한 이유는 Codex가 못해서가 아니라, 사람이 판단할 기준을 잃기 때문입니다. 실습 초반에는 결과 품질보다 "비교할 기준이 있는가"가 더 중요합니다.

## 7. 실습 체크리스트

- 별도 브랜치를 만들었습니다.
- `test` 또는 `pytest` 같은 최소 검증 명령을 확인했습니다.
- 필요한 런타임과 패키지 설치를 마쳤습니다.
- 샘플 저장소의 진입점과 테스트 위치를 확인했습니다.

## 정리

환경 세팅의 목적은 설치를 많이 하는 것이 아닙니다. 사람이 검토할 수 있는 변경 단위, 다시 돌려볼 수 있는 명령, 실패를 판단할 기준을 만드는 것입니다. 이 바닥이 단단해야 이후 장에서 Codex의 장점과 한계를 제대로 체감할 수 있습니다.
