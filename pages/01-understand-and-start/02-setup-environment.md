# 실습 환경 만들기

좋은 도구도 환경이 불안정하면 신뢰를 잃습니다. Codex를 쓸 때도 마찬가지입니다. 특히 초반에는 "도구가 틀렸다"는 문제와 "내 환경이 준비되지 않았다"는 문제가 자주 섞여 보이기 때문에, 실습 환경을 먼저 정리해두는 것이 중요합니다.

## Git 저장소와 브랜치 전략 준비

Codex와 협업할 때 Git은 단순한 백업 수단이 아니라 안전장치입니다. 작은 단위로 브랜치를 나누고, diff를 빠르게 검토하고, 실패했을 때 쉽게 되돌아갈 수 있어야 에이전트와의 협업 비용이 낮아집니다.

초반 실습에서는 복잡한 전략보다 단순한 규칙이 좋습니다.

- 메인 브랜치에서는 바로 작업하지 않습니다.
- 기능 추가, 버그 수정, 문서화는 각각 별도 브랜치로 분리합니다.
- 한 브랜치의 목적은 하나로 제한합니다.

브랜치 전략이 중요한 이유는 모델의 문제가 아니라 변경 범위를 사람이 통제할 수 있게 해주기 때문입니다. 에이전트가 한 번에 너무 많은 것을 건드렸을 때도, 브랜치와 diff가 작으면 판단이 쉬워집니다.

## CLI, IDE 확장, Codex app의 역할 구분

초보자는 종종 "어디서 쓰는 것이 정답인가"를 찾습니다. 하지만 실무에서는 도구별 역할 분담이 더 중요합니다.

- CLI는 코드베이스 탐색, 명령 실행, 테스트 검증, 반복 작업 자동화에 강합니다.
- IDE 확장은 편집 중인 파일과 주변 문맥을 빠르게 왕복하는 데 강합니다.
- Codex app은 스레드 관리, 병렬 작업, 작업 단위 분리에 강합니다.

같은 요청이라도 어디서 실행하느냐에 따라 체감이 달라집니다. 예를 들어 모듈 간 호출 흐름을 추적하고 실제 테스트를 돌릴 때는 CLI가 편하고, UI 문구를 다듬으며 컴포넌트 파일을 바로 보면서 수정할 때는 IDE가 편합니다.

## Python/Node 테스트 환경 준비

Codex와 협업할 때 테스트 환경은 선택이 아니라 기준점입니다. 에이전트가 변경을 제안했을 때 옳고 그름을 판정하는 가장 현실적인 방법은 테스트와 린트, 빌드 같은 자동 검증입니다.

Python 프로젝트라면 최소한 다음을 준비합니다.

- 가상환경 또는 의존성 관리 도구
- 테스트 실행 명령
- 린트 또는 포맷 검사 명령

Node 프로젝트도 같은 원리입니다.

- 패키지 매니저 설치 상태
- `test`, `lint`, `build` 스크립트
- 필요한 런타임 버전 고정

테스트 명령이 문서화되어 있지 않으면, Codex는 실행은 할 수 있어도 무엇을 기준으로 검증해야 하는지 흔들릴 수 있습니다.

## 샘플 프로젝트 내려받기

책 실습은 같은 예제 저장소를 여러 장에서 재사용하는 방식이 가장 좋습니다. 그래야 한 장에서는 탐색하고, 다음 장에서는 기능을 추가하고, 또 다음 장에서는 버그를 잡고, 마지막에는 자동화까지 연결할 수 있습니다.

샘플 프로젝트는 다음 조건을 만족하는 것이 좋습니다.

- Python 또는 Node 중 적어도 하나의 테스트 체인이 존재할 것
- 문서가 완벽하지 않을 것
- 일부러 작은 개선 포인트와 버그를 포함할 것
- 디렉터리 구조가 너무 크지도 너무 작지도 않을 것

좋은 예제 저장소는 도구를 돋보이게 만드는 무대입니다. 너무 단순하면 에이전트의 장점이 드러나지 않고, 너무 복잡하면 초반 학습 곡선이 가파릅니다.

## 정리

환경 세팅의 목적은 설치 자체가 아닙니다. 사람이 검토할 수 있는 변경 단위, 재현 가능한 실행 명령, 실패를 판정할 기준을 만드는 것이 핵심입니다. 이 기반이 있어야 이후 장에서 Codex의 장점과 한계를 분명하게 체감할 수 있습니다.
