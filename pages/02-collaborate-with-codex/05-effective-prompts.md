# 2-1. 잘 되는 프롬프트의 구조

Codex 프롬프트에서 중요한 것은 화려한 표현이 아닙니다. 목표, 범위, 제약, 출력 형식이 명확하냐가 거의 전부입니다. 이 네 요소가 분명하면 결과는 안정되고, 빠지면 출력은 그럴듯해 보여도 실무에서는 흔들립니다.

## 목차

- 목표, 범위, 제약, 출력 형식
- 구현이 아니라 명세를 전달하는 법
- 나쁜 요청과 좋은 요청 비교
- 반복 가능한 프롬프트 템플릿 만들기
- 정리

## 목표, 범위, 제약, 출력 형식

좋은 프롬프트는 보통 네 문장 블록으로 정리할 수 있습니다.

- 목표: 무엇을 바꾸려는가
- 범위: 어디까지 건드릴 수 있는가
- 제약: 무엇을 하면 안 되는가
- 출력 형식: 어떤 형태로 보고받고 싶은가

이 구조를 쓰면 Codex는 단순히 "무언가를 해보는" 대신, 사람이 검토하기 쉬운 결과를 만듭니다. 특히 범위와 제약이 빠지면, 모델은 문제를 해결하기 위해 더 넓은 변경을 시도할 가능성이 높아집니다.

## 구현이 아니라 명세를 전달하는 법

사용자가 구현 방법까지 미리 고정하면 오히려 Codex의 강점을 줄이는 경우가 있습니다. 더 좋은 방식은 요구사항과 불변조건을 명세로 주고, 구현은 Codex가 탐색하게 하는 것입니다.

예를 들어 "이 함수 안에 if문 하나 추가해"보다 "입력이 null이면 예외 대신 빈 결과를 반환해야 하고 기존 API는 유지해야 해"가 더 좋은 명세입니다. 전자는 방법을 강제하고, 후자는 결과와 제약을 정의합니다.

실무에서 명세 중심 요청이 중요한 이유는 구현 대안이 하나가 아닐 때입니다. 명세를 주면 Codex는 현재 코드베이스에 더 자연스러운 방법을 찾을 여지가 생깁니다.

## 나쁜 요청과 좋은 요청 비교

나쁜 요청의 특징은 대체로 같습니다.

- 범위가 지나치게 넓다.
- 기대 결과가 불분명하다.
- 검증 기준이 없다.
- 출력 형식이 없다.

좋은 요청은 반대로 작고 분명합니다. 예를 들어 "결제 실패 원인을 찾아서 고쳐줘"는 넓지만, "결제 실패 시 `payment_status`가 `null`일 때 500이 나는 문제를 재현하고, 원인을 찾고, 최소 수정과 회귀 테스트까지 포함해줘"는 훨씬 다루기 쉽습니다.

## 반복 가능한 프롬프트 템플릿 만들기

반복되는 업무는 프롬프트를 매번 새로 쓰지 않는 것이 좋습니다. 기능 추가, 버그 수정, 코드 탐색, 리뷰, 문서화처럼 자주 하는 작업은 템플릿으로 고정하면 품질 편차가 크게 줄어듭니다.

템플릿의 장점은 속도보다 일관성입니다. 어떤 팀원이 요청하든 결과 품질이 비슷해지고, 누락되는 정보가 줄어듭니다. 특히 여러 명이 함께 Codex를 쓸 때는 개인 감각보다 팀 공용 템플릿이 더 큰 효과를 냅니다.

## 정리

잘 되는 프롬프트는 길어서가 아니라 구조가 분명해서 잘 됩니다. 목표, 범위, 제약, 출력 형식만 일관되게 유지해도 Codex의 결과는 훨씬 실무 친화적으로 바뀝니다.
