# 2-2. 큰 작업을 쪼개는 법

Codex와 협업할 때 가장 자주 실패하는 순간은 "이 정도는 한 번에 되겠지"라고 생각하는 때입니다. 큰 작업은 인간에게도 어렵지만, 에이전트에게는 더 어렵습니다. 컨텍스트가 늘어나고, 검증 범위가 퍼지고, 잘못 바꿨을 때 되돌리기 비용이 커지기 때문입니다.

## 목차

- 탐색 -> 계획 -> 구현 -> 검증 루프
- 한 번에 큰 변경을 피해야 하는 이유
- PR 단위로 일 쪼개기
- 실패 비용이 낮은 작업 순서 설계
- 정리

## 탐색 -> 계획 -> 구현 -> 검증 루프

큰 작업은 네 단계 루프로 보는 것이 가장 안정적입니다.

1. 탐색: 관련 파일, 호출 흐름, 제약을 파악합니다.
2. 계획: 무엇을 어떤 순서로 바꿀지 정리합니다.
3. 구현: 실제 코드를 수정합니다.
4. 검증: 테스트, diff, 수동 확인으로 결과를 점검합니다.

문제는 이 네 단계를 한 번에 뭉개면 실패 원인이 안 보인다는 점입니다. 반대로 단계를 나누면 어디서 흔들렸는지 추적이 가능합니다.

## 한 번에 큰 변경을 피해야 하는 이유

한 번에 큰 변경을 하면 세 가지 문제가 동시에 생깁니다.

- 사람이 diff를 읽기 어려워집니다.
- 테스트 실패 시 원인 범위를 좁히기 어렵습니다.
- 좋은 변경과 나쁜 변경이 한 덩어리로 섞입니다.

에이전트 활용에서 중요한 원칙은 "능력이 크면 큰 작업을 맡겨도 된다"가 아니라 "검토 가능한 단위로 쪼개야 좋은 능력이 보인다"입니다.

## PR 단위로 일 쪼개기

PR은 단순한 병합 단위가 아니라 사고 단위입니다. 하나의 PR은 하나의 의도를 담아야 합니다. 버그 수정 PR, 문서 업데이트 PR, 리팩터링 PR, 테스트 보강 PR이 섞이기 시작하면 리뷰 품질이 급격히 떨어집니다.

Codex에게도 같은 원칙을 적용해야 합니다. "이 이슈 전체를 해결해"보다 "먼저 원인 분석만", "다음으로 최소 수정", "마지막으로 테스트 보강"처럼 PR 단위를 의식하며 나누면 결과가 안정됩니다.

## 실패 비용이 낮은 작업 순서 설계

좋은 작업 순서는 보통 위험이 낮은 곳에서 높은 곳으로 이동합니다.

- 먼저 읽기 전용 탐색을 합니다.
- 다음으로 작은 내부 변경을 합니다.
- 그 뒤에 공개 인터페이스나 데이터 구조 변경을 검토합니다.
- 마지막으로 문서와 예제를 맞춥니다.

이 순서가 중요한 이유는 앞단에서 얻은 정보가 뒤단의 실수를 줄이기 때문입니다. 특히 데이터 마이그레이션, API 변경, 인증 흐름 같은 고위험 수정은 충분한 탐색 이후에 들어가야 합니다.

## 정리

큰 작업을 잘게 쪼개는 것은 에이전트를 믿지 못해서가 아니라, 신뢰를 더 빠르게 쌓기 위한 방법입니다. 좋은 작업 분해는 리뷰 비용을 줄이고, 실패 원인을 분명하게 만들고, 모델 선택과 검증 전략까지 단순하게 만듭니다.
