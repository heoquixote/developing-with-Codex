# 2-3. AGENTS.md로 팀 규칙 심기

## 목차

1. 저장소 규칙을 Codex에게 지속적으로 전달하기
2. 코딩 컨벤션, 리뷰 기준, 금지 패턴 명시
3. 디렉터리별 세부 지침 설계
4. 팀별 규칙 문서 템플릿
정리

Codex와 협업하다 보면 같은 말을 반복하게 됩니다. "이 저장소는 테스트를 먼저 돌려야 한다", "이 폴더는 임의로 건드리면 안 된다", "리뷰는 버그 위험 중심으로 해달라" 같은 규칙들입니다. 이런 내용을 매번 프롬프트에 넣는 것은 비효율적이고, 시간이 갈수록 누락도 생깁니다. 이 문제를 줄이기 위한 가장 실용적인 방법이 `AGENTS.md`입니다.


## 1. 저장소 규칙을 Codex에게 지속적으로 전달하기

`AGENTS.md`의 역할은 도구 사용법을 설명하는 것이 아니라, 이 저장소에서 기대하는 협업 방식을 고정하는 것입니다. 즉, 일회성 요청이 아니라 지속적 컨텍스트를 제공하는 문서입니다.

좋은 `AGENTS.md`는 다음을 담습니다.

- 저장소의 구조와 목적
- 수정 시 지켜야 할 공통 원칙
- 테스트와 검증 기준
- 금지 또는 주의 패턴

이 문서가 있으면 Codex는 매번 새 작업을 할 때마다 팀의 기본 규칙을 더 안정적으로 따를 수 있습니다.


## 2. 코딩 컨벤션, 리뷰 기준, 금지 패턴 명시

팀 규칙은 "예쁘게 짜라"처럼 추상적이면 효과가 없습니다. 대신 구체적인 문장으로 적어야 합니다.

- 공개 API 시그니처 변경 금지
- 민감정보 로그 출력 금지
- 관련 테스트 없는 리팩터링 금지
- 리뷰는 성능보다 회귀 위험을 먼저 볼 것

이런 문장은 Codex에게도 분명하고, 사람에게도 분명합니다. 결국 좋은 팀 규칙은 사람과 에이전트 모두가 같은 방식으로 해석할 수 있어야 합니다.


## 3. 디렉터리별 세부 지침 설계

루트 수준의 규칙만으로는 부족한 경우가 많습니다. 프론트엔드와 백엔드는 위험 요소도 다르고, 데이터 파이프라인과 운영 스크립트도 기대하는 검증 방식이 다릅니다. 이럴 때는 디렉터리별 `AGENTS.md`가 유용합니다.

예를 들어 프론트엔드는 접근성, 디자인 시스템, 사용자 경험 일관성이 중요하고, 백엔드는 API 호환성, 마이그레이션, 로깅 정책이 중요할 수 있습니다. 같은 저장소 안에서도 위험이 다른 만큼 규칙도 분화되어야 합니다.


## 4. 팀별 규칙 문서 템플릿

팀에서 `AGENTS.md`를 만들 때는 다음 순서가 실용적입니다.

1. 저장소 목적과 주요 구성요소를 적습니다.
2. 공통 규칙을 5개 안팎으로 좁혀 적습니다.
3. 리뷰 기준을 우선순위 순으로 적습니다.
4. 디렉터리별 예외 규칙이 있으면 분리합니다.

규칙이 너무 많아지면 문서는 읽히지 않습니다. 핵심은 완전성보다 반복되는 실수를 줄이는 데 있습니다.


## 정리

`AGENTS.md`는 Codex를 더 똑똑하게 만드는 문서가 아니라, 팀의 기대를 더 일관되게 만드는 문서입니다. 결국 에이전트 협업의 품질은 모델 능력과 함께, 어떤 규칙을 얼마나 지속적으로 공급하느냐에 달려 있습니다.
