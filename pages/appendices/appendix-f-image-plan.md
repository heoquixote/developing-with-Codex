# Appendix F. 이미지와 도해 계획표

이 Appendix는 책 전체에서 어떤 장면에 이미지를 넣어야 이해도가 가장 크게 올라가는지를 정리한 작업표입니다. 특히 Codex 관련 주제는 화면 흐름, 작업 분리 구조, 승인 경계처럼 "글만으로는 한 번에 안 잡히는 개념"이 많기 때문에 우선순위를 두고 시각 자료를 준비하는 편이 좋습니다.

## 목차

- 우선순위 높은 이미지
- 우선순위 중간 이미지
- 우선순위 낮지만 있으면 좋은 이미지
- 제작 원칙

## 우선순위 높은 이미지

| 페이지 | 필요한 이미지 | 이미지 종류 | 이유 |
| --- | --- | --- | --- |
| 첫 번째 Codex 세션 | 첫 프롬프트 입력, diff 확인, 테스트 통과 화면 | 스크린샷 3종 | 책의 첫 성공 경험을 눈으로 보여줍니다. |
| Codex app과 worktree 기반 병렬 작업 | thread, worktree, branch 관계도 | 도식 | 가장 헷갈리는 개념을 한 번에 정리합니다. |
| MCP로 외부 도구 연결하기 | 사용자 -> Codex -> MCP -> 외부 도구 구조도 | 아키텍처 다이어그램 | MCP의 본질을 가장 빨리 이해시킵니다. |
| API에서 Codex 모델 직접 사용하기 | 요청/응답 시퀀스 다이어그램, 구조화 출력 예시 | 시퀀스 다이어그램 + 코드 캡처 | 제품 내장 관점을 실무적으로 보여줍니다. |
| 보안, 거버넌스, 선택 주제: Codex Security | 승인 경계, 권한 수준 매핑 도식 | 표 + 경계 다이어그램 | 실무 적용 시 판단 기준을 시각화합니다. |

## 우선순위 중간 이미지

| 페이지 | 필요한 이미지 | 이미지 종류 | 이유 |
| --- | --- | --- | --- |
| Codex를 어떤 도구로 봐야 하는가 | 코딩 보조도구 vs 코딩 에이전트 비교 | 비교 도표 | 책의 출발 개념을 선명하게 잡아줍니다. |
| 실습 환경 만들기 | CLI, IDE, App 역할 비교 | 3열 비교표 | 도구 선택 기준을 빠르게 이해시킵니다. |
| 모델 선택과 실행 모드 이해하기 | 속도·비용·정확도 매트릭스 | 비교 매트릭스 | 모델 선택 원리를 구조화합니다. |
| 큰 작업을 쪼개는 법 | 탐색 -> 계획 -> 구현 -> 검증 루프 | 프로세스 다이어그램 | 반복 가능한 작업 패턴을 설명하기 좋습니다. |
| AGENTS.md로 팀 규칙 심기 | 루트 규칙과 하위 디렉터리 규칙 계층 | 구조도 | 규칙 상속 개념을 직관적으로 보여줍니다. |
| IDE extension으로 페어 프로그래밍하기 | 실제 IDE 화면과 제안 diff | 스크린샷 | 사용자 체감 품질을 높입니다. |
| 관측성, 비용, 버전 변화 대응 | 운영 지표 대시보드 예시 | 대시보드 목업 | 운영 관점의 구체성을 올립니다. |

## 우선순위 낮지만 있으면 좋은 이미지

| 페이지 | 필요한 이미지 | 이미지 종류 | 이유 |
| --- | --- | --- | --- |
| GitHub PR 리뷰 자동화 | `@codex review`가 붙은 PR 화면 | 스크린샷 | 사용 흐름을 현실감 있게 보여줍니다. |
| 실전 프로젝트 1: 레거시 코드베이스 현대화 | 개선 전후 모듈 구조 비교 | before/after 구조도 | 결과 중심 설명에 도움이 됩니다. |
| 실전 프로젝트 2: 데이터/AI 엔지니어링 자동화 | 파이프라인 정리 전후 비교 | 파이프라인 다이어그램 | 데이터 실무 독자에게 설득력이 높습니다. |

## 제작 원칙

- 설명이 추상적인 페이지일수록 스크린샷보다 구조도가 더 중요합니다.
- UI가 자주 바뀌는 주제는 화면 전체보다 핵심 영역 캡처를 권장합니다.
- 모델명이나 버튼 위치보다 작업 원리를 보여주는 이미지가 책 수명이 깁니다.
- 같은 개념은 표, 도식, 화면 캡처 중 하나만 대표로 두고 중복을 줄입니다.

가장 먼저 제작할 이미지는 3장, 14장, 18장, 20장, 22장입니다. 이 다섯 군데만 먼저 채워도 독자의 이해 속도와 책의 완성도가 크게 올라갑니다.
