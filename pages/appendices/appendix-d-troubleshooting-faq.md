이 Appendix는 초반 실습과 팀 도입 과정에서 자주 부딪히는 문제를 빠르게 정리한 문서입니다.

## 목차

- Q1. Codex가 너무 넓게 수정합니다.
- Q2. 결과는 그럴듯한데 실무에 바로 쓰기 어렵습니다.
- Q3. 모델이 코드를 잘 읽지 못하는 것 같습니다.
- Q4. 승인 요청이 너무 자주 뜹니다.
- Q5. 테스트가 자꾸 실패하는데 원인이 코드인지 환경인지 모르겠습니다.
- Q6. Cloud 작업 결과를 바로 믿어도 되나요?
- Q7. 팀에 도입하려는데 어디서부터 시작해야 하나요?

## Q1. Codex가 너무 넓게 수정합니다.

대개 범위와 제약이 부족한 경우입니다. 목표만 주고 수정 범위를 주지 않으면, 모델은 문제 해결을 위해 더 넓은 변경을 시도할 수 있습니다.

## Q2. 결과는 그럴듯한데 실무에 바로 쓰기 어렵습니다.

출력 형식과 검증 기준이 부족했을 가능성이 큽니다. "수정해줘"보다 "테스트 포함, 변경 요약 포함, 관련 없는 리팩터링 금지"처럼 요구를 구조화해야 합니다.

## Q3. 모델이 코드를 잘 읽지 못하는 것 같습니다.

대개 모델 문제가 아니라 질문 범위가 너무 큽니다. 저장소 전체 설명을 요구하기보다 특정 기능 흐름이나 특정 모듈 설명으로 좁히는 편이 효과적입니다.

## Q4. 승인 요청이 너무 자주 뜹니다.

현재 승인 정책과 작업 유형이 맞지 않을 수 있습니다. 안전을 위해 막히는 것인지, 불필요하게 자주 막히는 것인지 구분해서 설정을 조정해야 합니다.

## Q5. 테스트가 자꾸 실패하는데 원인이 코드인지 환경인지 모르겠습니다.

실패 로그와 재현 단계를 먼저 분리해야 합니다. 코드 문제와 환경 문제를 섞어 보면 디버깅이 길어집니다. 가능한 한 동일 명령으로 반복 재현되는지부터 확인하는 것이 좋습니다.

## Q6. Cloud 작업 결과를 바로 믿어도 되나요?

아니요. Cloud 결과는 완성본이라기보다 검토 가능한 제안으로 보는 편이 안전합니다. diff, 테스트, 범위 적합성을 꼭 확인해야 합니다.

## Q7. 팀에 도입하려는데 어디서부터 시작해야 하나요?

처음부터 전면 도입하기보다 작은 반복 업무 하나를 골라 템플릿화하거나 리뷰 보조처럼 비차단형 활용부터 시작하는 편이 안정적입니다.
