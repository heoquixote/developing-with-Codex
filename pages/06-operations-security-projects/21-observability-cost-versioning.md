# 6-1. 관측성, 비용, 버전 변화 대응

## 목차

1. 모델 선택 전략
2. 로그, 이벤트, 텔레메트리 기본기
3. 비용을 잡는 운영 원칙
4. 잦은 제품 업데이트를 책과 팀에 반영하는 법
5. 운영 체크리스트
정리

도구를 도입하는 것보다 어려운 일은 도입 후 안정적으로 운영하는 것입니다. Codex도 마찬가지입니다. 초기에는 생산성 향상이 눈에 띄지만, 시간이 지나면 로그를 어떻게 볼지, 비용을 어떻게 통제할지, 모델과 제품 변화에 문서를 어떻게 맞출지가 더 중요해집니다.


## 1. 모델 선택 전략

운영 관점의 모델 선택은 "무엇이 가장 강한가"보다 "어떤 작업에 어떤 비용 구조가 맞는가"를 기준으로 해야 합니다. 탐색, 구현, 리뷰, 문서화, 자동화 각각이 요구하는 정확도와 지연 허용치가 다르기 때문입니다.

좋은 전략은 보통 단일 모델 고정보다 용도별 조합에 가깝습니다.


## 2. 로그, 이벤트, 텔레메트리 기본기

Codex 활용이 팀 단위로 커지면 질문이 생깁니다. 어떤 작업에 많이 쓰이는가, 어디서 실패가 많은가, 승인 요청은 어떤 유형에서 자주 나오는가 같은 운영 지표가 필요해집니다.

관측성의 목적은 감시가 아니라 개선입니다. 로그와 이벤트가 있어야 프롬프트를 바꿀지, 스킬을 만들지, 승인 정책을 조정할지 판단할 수 있습니다.


## 3. 비용을 잡는 운영 원칙

비용 통제는 단순히 작은 모델을 쓰는 것으로 해결되지 않습니다. 더 큰 비용 절감은 보통 작업 분해, 불필요한 반복 제거, 템플릿화, 구조화된 출력 도입에서 나옵니다.

실무적으로는 다음 원칙이 유용합니다.

- 탐색은 가볍게
- 고위험 판단은 깊게
- 반복 작업은 자동화
- 같은 질문은 템플릿화

즉, 비용은 모델 단가보다 워크플로 설계의 영향을 더 크게 받는 경우가 많습니다.

운영 지표는 최소한 아래 정도는 잡아두는 편이 좋습니다.

| 지표 | 왜 중요한가 | 실무 해석 |
| --- | --- | --- |
| 요청당 평균 비용 | 비용 급증 구간을 찾기 쉽습니다. | 특정 작업 유형이 과도하게 비싼지 봅니다. |
| 작업 완료율 | 실패 패턴을 빨리 볼 수 있습니다. | 프롬프트 문제인지 도구 문제인지 분리합니다. |
| 승인 요청 비율 | 위험 경계가 적절한지 알 수 있습니다. | 너무 높으면 자동화 이점이 줄고, 너무 낮으면 통제가 약할 수 있습니다. |
| 재시도 횟수 | 품질과 비용이 동시에 악화되는 신호입니다. | 반복 질문, 긴 컨텍스트, 도구 실패를 점검합니다. |


## 4. 잦은 제품 업데이트를 책과 팀에 반영하는 법

Codex처럼 변화가 빠른 도구를 다룰 때는 문서도 고정 자산이 아니라 운영 자산으로 봐야 합니다. 모델명, UI, 기능 경계가 자주 바뀔 수 있기 때문입니다.

그래서 책과 팀 문서에는 변하지 않는 원리와 바뀌는 표면을 분리해서 써야 합니다. 원리는 오래 가고, UI는 자주 갱신됩니다. 이 구분이 없으면 문서는 빨리 낡습니다.


## 5. 운영 체크리스트

- 작업 유형별로 쓰는 모델 기준을 정리했습니다.
- 비용과 실패율을 함께 볼 수 있는 로그 지표를 정했습니다.
- 반복 질문은 템플릿 또는 스킬로 흡수하고 있습니다.
- 자주 바뀌는 UI 설명과 오래 가는 원리를 문서에서 분리했습니다.


## 정리

운영의 핵심은 도입 초반의 효율을 오래 유지하는 것입니다. 관측성, 비용, 버전 변화 대응은 모두 그 지속성을 위한 장치입니다.
