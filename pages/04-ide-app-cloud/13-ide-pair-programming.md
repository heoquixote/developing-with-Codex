정리

IDE 안에서 Codex를 쓰는 경험은 CLI와 다릅니다. CLI가 작업 단위와 검증 흐름 중심이라면, IDE는 현재 보고 있는 코드와 주변 맥락에 더 밀착되어 있습니다. 그래서 탐색과 구현 사이의 왕복이 빠르고, "바로 이 파일에서 지금 무엇을 바꿔야 하는가"에 집중하기 좋습니다.


## 1. 에디터 옆에서 Codex 쓰기

IDE에서 Codex를 잘 쓰는 핵심은 대화를 길게 끌지 않는 것입니다. 현재 열려 있는 파일, 선택 영역, 에러 표시, 테스트 실패 위치처럼 이미 화면에 보이는 맥락을 적극적으로 활용해야 합니다.

이 환경에서는 긴 배경 설명보다 짧고 구체적인 요청이 효과적입니다. "이 컴포넌트의 조건 분기를 단순화해줘", "이 훅이 왜 무한 렌더링을 일으키는지 설명해줘"처럼 현재 파일과 직접 연결된 요청이 잘 맞습니다.

예를 들어 아래 같은 요청은 IDE 맥락과 잘 맞습니다.

```text
현재 열려 있는 React 컴포넌트에서 로딩 상태 분기가 중복됩니다.
중복을 줄이되 UI 동작은 바꾸지 말고, 변경 이유를 diff 기준으로 설명해주세요.
```


## 2. 파일 단위 수정과 리뷰

IDE 기반 협업의 강점은 수정과 리뷰가 같은 화면에서 이어진다는 점입니다. Codex가 제안한 변경을 바로 diff로 보고, 인접 코드와 비교하고, 필요한 경우 같은 자리에서 추가 요청을 이어갈 수 있습니다.

실무에서는 이 점이 중요합니다. 작은 범위의 UI 수정, 함수 정리, 타입 보강, 문구 수정 같은 작업은 IDE에서 가장 높은 생산성을 보이는 경우가 많습니다.


## 3. 에디터 컨텍스트를 활용한 대화

IDE 확장의 진짜 장점은 텍스트 입력량이 아니라 컨텍스트 전달 비용을 줄여준다는 점입니다. 어떤 파일을 보고 있는지, 어떤 줄이 문제인지, 어떤 에러가 떠 있는지 이미 공유되기 때문에 요청이 더 짧고 정밀해집니다.

다만 이 장점은 역으로 함정이 되기도 합니다. 현재 파일만 보고 큰 구조를 놓치기 쉽기 때문입니다. 따라서 파일 단위 수정이 구조적 영향으로 이어질 가능성이 보이면, CLI나 더 넓은 탐색 모드로 전환하는 판단이 필요합니다.


## 4. 화면에서 꼭 확인할 포인트

IDE 기반 협업은 화면 맥락을 활용하는 대신, 실제로 어떤 정보가 공유되고 있는지 놓치기 쉽습니다. 아래 항목은 항상 눈으로 확인하는 편이 좋습니다.

- 현재 열려 있는 파일과 선택 영역이 맞는지
- 에러 패널이나 테스트 실패 위치가 최신 상태인지
- 제안 diff가 현재 편집 중인 파일 범위를 넘지 않는지
- 자동완성과 Codex 제안이 섞여 보일 때 출처를 구분할 수 있는지

실무에서는 이 확인만 잘해도 "의도하지 않은 파일 수정"이나 "오래된 에러 기준으로 잘못 고친 변경"을 꽤 줄일 수 있습니다.


## 5. IDE 기반 작업의 장단점

장점은 분명합니다.

- 빠른 왕복
- 파일 단위 집중
- 즉시 시각적 검토
- 편집 흐름과 자연스럽게 결합

단점도 분명합니다.

- 저장소 전체 관점이 약해질 수 있음
- 긴 작업 흐름 관리에는 불리함
- 여러 작업을 병렬로 분리하기 어려움

결국 IDE는 가장 일상적인 협업 공간이지만, 모든 작업에 최적은 아닙니다.


## 6. 자주 하는 실수

- 저장소 전체 영향이 있는 변경도 IDE 맥락만 믿고 처리합니다.
- 선택 영역이 잘못된 상태에서 수정 요청을 보냅니다.
- 테스트 실패 화면이 오래된 상태인지 확인하지 않습니다.
- 빠른 왕복이 가능하다는 이유로 diff 검토를 생략합니다.


## 정리

IDE extension은 Codex를 "바로 옆에 앉아 있는 페어 프로그래머"처럼 느끼게 만드는 환경입니다. 빠른 수정과 짧은 탐색에는 강하지만, 넓은 범위의 작업 설계와 병렬 분할은 다른 실행 환경과 함께 써야 더 큰 힘을 냅니다.
