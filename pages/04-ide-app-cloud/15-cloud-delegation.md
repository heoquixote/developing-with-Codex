# 4-3. Codex cloud로 긴 작업 위임하기

## 목차

1. 오래 걸리는 작업 맡기기
2. 병렬 브랜치 실험
3. 결과 diff를 검토하고 병합하기
4. local flow와 cloud flow의 역할 분담
정리

모든 작업을 로컬에서 직접 처리할 필요는 없습니다. 오래 걸리는 탐색, 대안 비교, 대규모 문서 초안 작성, 병렬 실험처럼 당장 화면 앞에서 기다릴 필요가 없는 작업은 Cloud 위임이 더 잘 맞습니다.


## 1. 오래 걸리는 작업 맡기기

Cloud 위임은 "시간이 걸려도 되는 일"에 잘 맞습니다. 예를 들어 전체 코드베이스에서 deprecated API 사용 지점을 찾거나, 여러 설계 대안을 비교하거나, 긴 문서 초안을 만드는 작업은 로컬 인터랙션보다 위임 효율이 높을 수 있습니다.

핵심은 위임 전에 산출물을 분명히 정하는 것입니다. 무엇을 조사할지, 어떤 형식으로 결과를 받을지, 어떤 기준으로 비교할지를 먼저 고정해야 결과 해석이 쉬워집니다.


## 2. 병렬 브랜치 실험

Cloud의 또 다른 장점은 병렬 실험입니다. 하나의 요구사항에 대해 서로 다른 설계나 구현 방향을 나눠서 시도해볼 수 있습니다. 실무에서는 이것이 큰 가치가 있습니다. 논쟁을 길게 하기보다, 작은 diff를 두세 개 받아 비교하는 편이 더 빠른 경우가 많기 때문입니다.

다만 실험 브랜치는 "아이디어 발산"보다 "비교 가능한 대안"이어야 합니다. 기준이 없으면 브랜치 수만 늘고 판단은 더 어려워집니다.


## 3. 결과 diff를 검토하고 병합하기

Cloud 위임의 핵심은 생성이 아니라 수렴입니다. 결국 결과는 사람이 검토하고 선택해야 합니다. 따라서 diff를 받을 때는 다음 질문이 중요합니다.

- 요청한 범위 안에서 작업했는가
- 대안 간 차이가 무엇인가
- 테스트와 문서가 함께 반영됐는가
- 지금 병합해도 되는가, 아니면 로컬에서 다듬어야 하는가

Cloud 위임이 잘 되려면 결과를 "완성본"이 아니라 "검토 가능한 제안"으로 받아들이는 태도가 필요합니다.


## 4. local flow와 cloud flow의 역할 분담

정리하면 역할은 대체로 이렇습니다.

- 로컬: 즉시 수정, 세밀한 확인, 테스트 중심 작업
- Cloud: 오래 걸리는 조사, 병렬 대안 생성, 큰 초안 작성

이 구분이 명확할수록 작업 흐름이 단순해집니다. 모든 것을 로컬에서 하려 하면 기다림이 길어지고, 모든 것을 Cloud에 넘기면 통제력이 약해집니다.


## 정리

Cloud 위임은 더 똑똑한 작업 방식이라기보다 더 나은 큐 관리 방식입니다. 지금 붙잡고 있을 필요가 없는 일을 분리해두면, 로컬에서는 더 중요한 판단과 검증에 집중할 수 있습니다.
