# 3-4. 리팩터링, 문서화, 테스트 자동화

기능 추가와 버그 수정만으로는 코드베이스 품질이 유지되지 않습니다. 시간이 지나면 중복이 쌓이고, 오래된 API가 남고, 테스트는 부족해지고, 문서는 구현과 멀어집니다. Codex는 이런 유지보수형 작업에서 특히 유용합니다. 반복적이지만 귀찮은 일을 빠르게 정리해주기 때문입니다.

## 목차

- 함수 분리와 모듈 정리
- deprecated API 마이그레이션
- 테스트 케이스 보강
- README, 주석, 운영 문서 정리
- 정리

## 함수 분리와 모듈 정리

리팩터링의 목표는 코드를 더 똑똑하게 만드는 것이 아니라, 더 읽기 쉽게 만드는 것입니다. 함수가 너무 길거나, 하나의 모듈이 너무 많은 책임을 가지거나, 데이터 변환과 부수효과가 뒤섞여 있으면 이후 변경 비용이 커집니다.

Codex에게 리팩터링을 맡길 때는 동작 변경 금지, 공개 인터페이스 유지, 테스트 보존 같은 제약을 함께 줘야 합니다. 그렇지 않으면 "더 나아 보이는 구조"를 향한 과감한 변경이 들어갈 수 있습니다.

## deprecated API 마이그레이션

deprecated API 정리는 미루기 쉬운 작업이지만, 늦게 할수록 비용이 커집니다. 어느 순간 라이브러리 업그레이드가 막히고, 경고가 누적되며, 관련 코드가 여러 곳에 퍼져 있으면 한 번에 큰 정리가 필요해집니다.

Codex는 사용 지점을 찾고, 패턴을 비교하고, 반복 수정안을 만드는 데 강합니다. 다만 중요한 것은 새 API로 바꾸는 과정에서 의미가 정말 동일한지 검증하는 것입니다. 이름이 같아 보여도 동작 차이가 있을 수 있기 때문입니다.

## 테스트 케이스 보강

테스트가 부족한 코드베이스는 변경 자체보다 변경 후 불안이 더 큽니다. 테스트 보강은 단순히 커버리지를 올리는 작업이 아니라, 팀이 변경을 수용할 수 있는 자신감을 높이는 작업입니다.

좋은 테스트 보강은 다음을 포함합니다.

- 핵심 경로의 정상 케이스
- 경계값과 실패 케이스
- 이전에 발생한 회귀 패턴

Codex는 이런 테스트 초안을 빠르게 제안할 수 있지만, 어떤 테스트가 유지 가치가 높은지는 사람의 도메인 판단이 필요합니다.

## README, 주석, 운영 문서 정리

문서화는 코드보다 덜 중요해 보이지만, 협업 비용에는 직접적인 영향을 줍니다. 특히 에이전트를 사용하는 팀일수록 문서는 더 중요합니다. 사람뿐 아니라 도구도 문서를 컨텍스트로 읽기 때문입니다.

README에는 보통 설치와 실행만 적혀 있지만, 실무에서는 다음이 더 중요합니다.

- 이 저장소가 해결하는 문제
- 자주 쓰는 명령
- 실패 시 확인할 지점
- 변경 시 주의할 규칙

주석과 운영 문서도 같은 원리입니다. 없어서 문제인 것보다, 낡아서 더 문제인 경우가 많습니다.

## 정리

리팩터링, 테스트, 문서화는 화려한 작업은 아니지만 생산성을 지탱하는 기반입니다. Codex는 이런 기반 작업을 덜 지루하게, 더 빠르게, 더 반복 가능하게 만들어줍니다.
