# 3-3. 버그 수정과 디버깅

디버깅은 Codex 활용에서 체감 가치가 큰 영역입니다. 이미 문제가 드러나 있고, 재현 단계와 오류 신호가 존재하며, 정답 검증도 비교적 명확하기 때문입니다. 반대로 이 조건이 없는 디버깅은 사람에게도 어렵고 에이전트에게도 어렵습니다.

## 목차

- 오류 재현 단계 정리하기
- 로그와 stack trace로 root cause 찾기
- high-confidence change만 적용하기
- 회귀 테스트 추가하기
- 정리

## 오류 재현 단계 정리하기

좋은 버그 수정은 원인 분석보다 먼저 재현 정리에서 시작합니다. 재현이 불명확하면 수정도 검증도 흐려집니다.

재현 단계에는 최소한 다음이 들어가야 합니다.

- 어떤 입력이나 상태에서 발생하는가
- 실제로 무엇이 일어나는가
- 기대한 정상 결과는 무엇인가

Codex에게도 이 세 가지를 넘겨야 문제 공간을 좁힐 수 있습니다.

## 로그와 stack trace로 root cause 찾기

디버깅에서 로그와 stack trace는 단서가 아니라 거의 지도에 가깝습니다. 중요한 것은 메시지를 읽는 것이 아니라, 어느 계층에서 실패가 시작됐는지 찾는 것입니다.

예를 들어 예외가 컨트롤러에서 터졌더라도 원인은 데이터 변환, null 처리, 외부 API 응답, 비동기 흐름 중 어딘가에 있을 수 있습니다. 따라서 Codex에게 로그를 보여줄 때는 한 줄 메시지보다 앞뒤 문맥과 재현 단계가 함께 있어야 합니다.

## high-confidence change만 적용하기

버그를 빨리 고치고 싶은 마음에 여러 가능성을 한 번에 수정하는 경우가 많습니다. 하지만 이렇게 하면 실제 원인이 무엇이었는지 사라지고, 부작용이 늘어납니다. Codex와 협업할 때는 특히 `high-confidence change` 원칙이 중요합니다.

의미는 단순합니다.

- 재현된 원인과 직접 연결된 수정만 한다.
- 증거 없는 리팩터링을 섞지 않는다.
- 필요한 경우 진단 로그와 테스트를 먼저 추가한다.

이 원칙을 지키면 수정 자체보다도 이후 리뷰와 회고가 쉬워집니다.

## 회귀 테스트 추가하기

버그 수정이 끝났다면 같은 문제가 다시 생기지 않도록 테스트를 추가해야 합니다. 이것이 없으면 수정은 일회성 패치에 그칠 수 있습니다.

좋은 회귀 테스트는 버그를 재현하는 최소 조건을 담습니다. 너무 넓게 쓰면 유지보수가 힘들고, 너무 좁게 쓰면 실제 실패 패턴을 놓칠 수 있습니다. Codex는 테스트 초안을 빠르게 만드는 데 강하지만, 어떤 조건을 경계값으로 삼을지는 사람이 판단해야 합니다.

## 정리

버그 수정의 핵심은 영리한 추측이 아니라 재현, 근거, 최소 수정, 회귀 방지입니다. Codex는 이 과정에서 속도를 올려주지만, 흐름 자체는 사람이 설계해야 가장 안정적입니다.
