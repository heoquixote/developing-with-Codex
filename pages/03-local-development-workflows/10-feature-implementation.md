# 3-2. 기능 추가 실습

Codex에게 기능 추가를 잘 맡기려면 구현을 시키기 전에 요구사항을 정리해야 합니다. 실무에서 기능 추가가 실패하는 가장 흔한 이유는 모델이 코드를 못 써서가 아니라, 사람이 요구사항을 충분히 구조화하지 못해서입니다.

## 목차

- 요구사항을 spec으로 바꾸기
- 최소 구현 버전 만들기
- 테스트까지 포함해 수정시키기
- 문서와 예제 업데이트까지 마무리하기
- 정리

## 요구사항을 spec으로 바꾸기

요구사항을 그대로 전달하면 대개 너무 넓습니다. "검색 기능을 개선해줘"는 목표가 모호합니다. 반면 "검색어가 비어 있을 때 전체 조회를 하지 말고, 최소 길이 2자를 요구하며, 기존 API 응답 스키마는 유지해줘"는 명세에 가깝습니다.

명세로 바꿀 때는 다음을 분리해서 적는 것이 좋습니다.

- 사용자 관점의 기대 동작
- 변경 가능한 범위
- 유지해야 할 불변조건
- 검증 기준

## 최소 구현 버전 만들기

기능 추가는 처음부터 완성형을 노리기보다 최소 구현 버전으로 시작하는 것이 안전합니다. 이유는 두 가지입니다. 첫째, diff가 작아집니다. 둘째, 동작 검증과 피드백이 빨라집니다.

최소 구현은 흔히 다음 질문으로 정해집니다.

- 이 기능의 핵심 가치가 무엇인가
- 이번 변경에서 반드시 들어가야 하는 것은 무엇인가
- 나중 단계로 미뤄도 되는 것은 무엇인가

Codex에게도 이 구분을 먼저 주면, 과도한 범위 확장을 줄일 수 있습니다.

## 테스트까지 포함해 수정시키기

기능 추가에서 테스트는 부록이 아닙니다. 구현 자체의 일부입니다. 테스트가 없으면 "동작했다"는 주장이 검증되지 않고, 이후 변경에서 회귀를 막기도 어렵습니다.

Codex에게 기능 추가를 맡길 때는 구현과 테스트를 한 요청 안에 묶는 편이 좋습니다. 그러면 모델이 처음부터 검증 가능한 단위로 생각하게 됩니다. 특히 경계 조건, 실패 케이스, 기존 동작 보존 여부를 테스트로 남기면 이후 유지보수 비용이 크게 줄어듭니다.

## 문서와 예제 업데이트까지 마무리하기

실무에서 기능은 코드만으로 끝나지 않습니다. README, API 예시, 운영 문서, 샘플 호출 예시까지 함께 바뀌어야 팀이 따라올 수 있습니다. Codex는 이 후처리 작업에도 강합니다.

중요한 점은 문서를 "있으면 좋은 것"이 아니라 변경 범위의 일부로 보는 것입니다. 기능을 추가했는데 사용 방법이 문서에 반영되지 않으면, 구현보다 문서가 먼저 낡기 시작합니다.

## 정리

기능 추가를 잘하는 핵심은 거창한 구현이 아니라 명세, 최소 구현, 테스트, 문서화의 순서를 지키는 것입니다. Codex는 이 순서를 따를 때 가장 실무적으로 유용해집니다.
