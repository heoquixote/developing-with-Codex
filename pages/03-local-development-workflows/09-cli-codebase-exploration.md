정리

낯선 저장소를 만났을 때 가장 먼저 필요한 능력은 "빨리 수정하기"가 아니라 "어디를 봐야 하는지 아는 것"입니다. CLI는 이 과정에서 매우 강력합니다. 에디터보다 빠르게 범위를 좁히고, Codex에게도 명확한 탐색 단서를 줄 수 있기 때문입니다.


## 1. 모듈 구조 파악

첫 단계는 저장소 지도를 그리는 것입니다. 디렉터리 구조, 엔트리포인트, 테스트 위치, 설정 파일, 빌드 스크립트부터 빠르게 확인해야 합니다.

이 단계에서 좋은 질문은 보통 다음과 같습니다.

- 애플리케이션 시작점이 어디인가
- 도메인별 디렉터리가 어떻게 나뉘는가
- 테스트와 운영 설정은 어디에 있는가

Codex에게도 이 수준의 지도를 먼저 설명하게 하면, 이후 수정 요청의 정확도가 올라갑니다.


## 2. 레거시 코드 설명 받기

레거시 코드는 대개 주석이 부족해서 어려운 것이 아니라, 의도가 코드에 분산되어 있어서 어렵습니다. 같은 기능이 여러 파일에 걸쳐 있고, 예전 제약 때문에 생긴 우회 로직이 남아 있으며, 이름만 봐서는 실제 역할이 보이지 않는 경우가 많습니다.

이럴 때 Codex에게 "이 모듈이 왜 이렇게 생겼는지 추정해줘"라고 묻는 것보다, "입력부터 출력까지 흐름을 따라 핵심 분기와 의존성을 설명해줘"라고 요청하는 편이 낫습니다. 구조적 설명을 받아야 나중에 수정 포인트도 판단할 수 있기 때문입니다.


## 3. 특정 기능 흐름 추적하기

실무에서는 코드베이스 전체를 이해하는 것보다 특정 기능 하나를 끝까지 추적하는 일이 더 많습니다. 로그인, 주문 생성, 리포트 배치, 알림 발송처럼 하나의 기능 흐름을 잡고 들어가면 관련 파일과 데이터 구조가 빠르게 좁혀집니다.

좋은 기능 추적의 핵심은 "진입점 -> 중간 처리 -> 저장 또는 출력" 순서로 보는 것입니다. 이 순서를 기준으로 Codex에게 관련 파일과 호출 관계를 정리하게 하면, 사람도 리뷰 포인트를 더 쉽게 잡을 수 있습니다.


## 4. 문서가 없는 코드 이해하기

문서가 없는 코드를 이해할 때 가장 위험한 태도는 "이름이 이러니 아마 이런 역할일 것"이라는 추측입니다. 추측은 빠르지만 자주 틀립니다. 더 안전한 방법은 실제 호출 경로, 데이터 변환, 테스트 케이스, 예외 처리 분기를 근거로 해석하는 것입니다.

Codex는 이런 근거 정리에 특히 유용합니다. 단, 설명을 그대로 믿기보다 실제 파일과 함께 교차 검증하는 습관이 중요합니다. 설명은 출발점이지 최종 진실이 아닙니다.


## 정리

CLI 기반 탐색은 단순히 터미널을 잘 쓰는 기술이 아닙니다. 코드베이스를 더 작은 질문으로 분해하고, Codex가 탐색할 수 있는 경로를 분명하게 만드는 기술입니다. 이 능력이 있어야 기능 추가와 디버깅도 안정적으로 이어집니다.
